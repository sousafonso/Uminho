
A elicitação de requisitos, também conhecida como descoberta ou aquisição de requisitos, é essencial para compreender as necessidades e expetativas das partes interessadas relativamente a um sistema. Este processo envolve:

- **1. Estudar o Domínio:** Obter conhecimento sobre o ambiente no qual o sistema irá operar;

- **2. Identificar Fontes:** Determinar os indivíduos ou artefactos que fornecem informações valiosas;

- **3. Consultar Partes Interessadas:** Envolver as partes interessadas para recolher as suas contribuições;

- **4. Selecionar Técnicas:** Escolher métodos apropriados para elicitar requisitos;

- **5. Elicitar Requisitos:** Capturar a informação necessária para o desenvolvimento do sistema.

As atividades de elicitação estendem-se ao longo de todo o ciclo de vida do sistema, garantindo flexibilidade e adaptabilidade às mudanças.

## Identificação de Partes Interessadas

As partes interessadas são uma fonte primária de informação durante a elicitação. Incluem indivíduos ou entidades com interesses legítimos no sistema, direta ou indiretamente afetados pelo seu desenvolvimento. Os pontos-chave incluem:

- **1. Papéis:**
	- As partes interessadas podem ocupar múltiplos papéis (por exemplo, diretor da empresa, contabilidade);
	- Os papéis devem ser identificados e documentados para clarificar responsabilidades.
- **2. Tipos de Partes Interessadas:**
	- **Utilizadores:** Operadores diretos do sistema (por exemplo, condutores, passageiros);
	- **Clientes:** Entidades que encomendam e pagam pelo desenvolvimento do sistema, frequentemente distintos dos utilizadores;
	- **Consumidores:** Consumidores finais que compram ou utilizam o produto;
	- **Segmentos de Mercado:** Grupos de consumidores com necessidades ou comportamentos partilhados;
	- **Partes Interessadas Negativas:** Indivíduos que se opõem ao desenvolvimento do sistema, potencialmente devido a razões pessoais ou políticas.
- **3. Métodos de Identificação:**
	- Consultar clientes;
	- Analisar organogramas;
	- Examinar produtos similares;
	- Rever o contexto do sistema.

Compreender a diversidade das partes interessadas, como incapacidades ou níveis de literacia, é crucial para criar sistemas inclusivos.

## Técnicas para Elicitação de Requisitos

Podemos categorizá-las em três: _técnicas individuais_, _técnicas de grupo_ e _técnicas baseadas em artefactos:_

### Técnicas Individuais

- **1. Entrevistas:**
	- Interações estruturadas ou semi-estruturadas com partes interessadas para recolher informações detalhadas;
	- Passos essenciais incluem identificar entrevistados, preparar questões, conduzir a entrevista e resumir descobertas.
- **2. Inquéritos:**
	- Recolher dados de um público mais amplo usando questionários;
	- Eficazes quando bem concebidos, com questões claras e concisas.
- **3. Introspeção:**
	- Os analistas baseiam-se na sua compreensão do domínio para inferir requisitos;
	- Útil como ponto de partida mas não deve substituir a contribuição direta das partes interessadas.

### Técnicas de Grupo

- **1. Brainstorming:**
	- Gerar uma ampla gama de ideias sem julgamento;
	- Eficaz para explorar possibilidades e consolidá-las em insights acionáveis.

### Técnicas Baseadas em Artefactos

- **1. Análise de Domínio:**
	- Rever documentação existente e estudar sistemas similares para extrair requisitos;
	- Ajuda a construir conhecimento do domínio e identificar padrões comuns.
- **2. Prototipagem:**
	- Criar modelos preliminares para capturar requisitos iterativamente;
	- Útil quando as partes interessadas não conseguem articular necessidades detalhadas inicialmente.
- **3. Personas:**
	- Desenvolver perfis fictícios para representar utilizadores-alvo;
	- As personas destacam objetivos, comportamentos e contextos dos utilizadores para garantir que o sistema se alinha com as suas necessidades.
