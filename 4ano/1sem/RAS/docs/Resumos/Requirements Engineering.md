A engenharia de requisitos determina o que o sistema deve fazer para satisfazer as necessidades dos utilizadores, e não como deve ser construído. É o estudo de um problema antes de qualquer ação de design ou implementação. É o processo de descobrir, analisar, documentar e verificar os serviços e restrições.
- **Âmbito:** Transforma descrições informais do mundo real em especificações formais do sistema;
- **Objetivo:** A engenharia de requisitos visa estudar o domínio do problema minuciosamente antes de iniciar o design ou implementação;
- **Foco:** Aborda objetivos e restrições do mundo real para sistemas de software.

Os objetivos principais da engenharia de requisitos são:

- Garantir que todos os requisitos relevantes sejam **identificados**, **compreendidos** e **documentados** ao nível apropriado de detalhe;
- Alcançar **acordo** sobre os requisitos **entre as partes interessadas**;
- Documentar requisitos usando **formatos** e **modelos** **definidos**.

A Engenharia de Requisitos foca-se no **"o quê"** — o que o sistema deve fazer para satisfazer as necessidades dos utilizadores — **e não no "como"** — a tecnologia ou a arquitetura que será usada para o construir.

## Atividades na Engenharia de Requisitos

O processo de engenharia de requisitos consiste em sete atividades principais:

- **1. Início:**
	- O objetivo é realizar uma recolha de informação em largura para compreender a visão do cliente e o potencial retorno do investimento
- **2. Elicitação:**
	- Foca-se em recolher requisitos das partes interessadas;
	- Envolve técnicas como entrevistas, inquéritos, etnografia, prototipagem e análise de cenários.
- **3. Elaboração:**
	- Organiza e analisa os requisitos elicitados;
	- Garante que os requisitos são coerentes, não-contraditórios, completos e livres de ambiguidade.
- **4. Negociação:**
	- Aborda conflitos entre partes interessadas relativamente aos requisitos;
	- Utiliza técnicas de priorização para decidir quais requisitos devem ser implementados primeiro.
- **5. Documentação:**
	- Produz um documento de referência para as fases de desenvolvimento;
	- A documentação é categorizada em requisitos de utilizador (expetativas gerais) e requisitos de sistema (especificações detalhadas).
- **6. Validação:**
	- Garante que os requisitos documentados correspondem ao que o cliente deseja;
	- Realizada através de inspeções e revisões.
- **7. Gestão:**
	- Lida com mudanças nos requisitos ao longo do processo de desenvolvimento;
	- Identifica mecanismos para avaliar e incorporar mudanças sem perturbações significativas.

## Desafios e Problemas na Engenharia de Requisitos

- **1. Dificuldades de Comunicação:**
	- As partes interessadas podem ter dificuldade em articular as suas necessidades claramente;
	- Os engenheiros frequentemente interpretam mal os requisitos do cliente, levando a soluções incorretas.
- **2. Natureza Dinâmica dos Requisitos:**
	- Os requisitos mudam frequentemente durante o desenvolvimento;
	- Por exemplo, projetos da Microsoft reportadamente experienciam uma mudança de 30% nos requisitos até ao final da fase de desenvolvimento.
- **3. Uso de Jargão:**
	- A linguagem técnica pode criar mal-entendidos entre engenheiros e partes interessadas.
- **4. Complexidade da Tarefa:**
	- A engenharia de requisitos é um dos aspetos mais desafiantes do desenvolvimento de software;
	- Nenhuma técnica de elicitação única é suficiente para todos os sistemas, exigindo que os engenheiros se adaptem com base no contexto e complexidade do projeto.

Para abordar os desafios, técnicas como observação de utilizadores, envolvimento de partes interessadas e gestão flexível de mudanças são recomendadas. Uma compreensão clara do domínio do problema é essencial para evitar soluções desalinhadas.
